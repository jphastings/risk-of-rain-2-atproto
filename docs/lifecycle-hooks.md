# Lifecycle hooks

Where and how the mod listens for "the run state meaningfully changed."
All names are RoR2-side. The vast majority are vanilla `Class.onXGlobal`
static events; a few use MonoMod `On.RoR2.*` hooks where no vanilla event exists.

## Stack

- Unity Mono, **BepInEx 5.4.x**.
- Hooks via **MonoMod** (`On.RoR2.*` and the `Class.onXGlobal` static events
  that ship in `RoR2.dll`). `MMHOOK_RoR2.dll` is regenerated by
  [HookGenPatcher][hookgen] on game update; the mod depends on it.
- HarmonyX is *not* used. It works, but it's non-idiomatic for RoR2 and
  loses us the on-update resilience HookGenPatcher gives MonoMod hooks.
- R2API is **not** a hard dependency for telemetry. We can ship without it.

## Emission policy

A "snapshot" is a single `putRecord` to `me.byjp.pesos.ror2.run` (rkey stable
across the run â€” see [`atproto-records.md`](atproto-records.md)).

The mod runs an emission loop driven by two signals:

1. **Dirty bit** â€” an event flips it on. Examples: stage advance, boss kill,
   item pickup, player death. While dirty, the loop is allowed to emit.
2. **Throttle** â€” at most one emission per 60 s (configurable). Plus an
   immediate emit on terminal events (game over) regardless of throttle.

Why throttle: RoR2 fires `Inventory.onInventoryChanged` dozens of times per
stage, and the user only cares about state every minute or so. Throttling
keeps PDS write volume sane for the user's storage allowance and reduces our
backoff exposure.

There's no `Stage.onStageStartGlobal` analogue in StS2's `SaveManager.Saved`
catch-all, so the throttled timer also acts as our safety net: even with no
events, if the run is in-progress we'll emit a fresh snapshot at the throttle
interval so the user's PDS reflects "still alive on Petrichor V, 18 minutes in."

## Hook catalogue

Grouped by the policy decision they trigger. **DIRTY** sets the dirty bit;
**FLUSH** forces an immediate emission. **DBOUNCE** sets dirty but coalesces
multiple firings within ~250ms so a pickup-burst doesn't queue multiple flushes.

### Lifecycle (the spine)

| Event | Authority | Action | Notes |
|---|---|---|---|
| `RoR2.RoR2Application.onLoad` | client | wire hooks | One-shot. Safe register-hooks point. |
| `RoR2.Run.onRunStartGlobal(Run)` | all peers | **start tracking**, FLUSH | Read `Run.instance.seed`, `runStartTimeUtc`. Re-fires on save-load via ProperSave-like flows; suppress dupes by tracking `Run.instance` identity. |
| `RoR2.Run.onServerGameOver(Run, GameEndingDef)` | server | finalize, FLUSH | Win/loss authoritative. Pair with `RunReport.Generate(Run.instance, def)` for end-of-run snapshot. |
| `RoR2.Run.onClientGameOver(Run, RunReport)` | client | finalize, FLUSH | For non-host peers. |
| `RoR2.Run.onRunDestroyGlobal(Run)` | all peers | clear tracking | Return-to-menu and post-game-over teardown. |

### ProperSave bridge (soft dependency)

When the [ProperSave][propersave] mod is present, we subscribe to two
of its events to preserve `startTimeUtc` across save-and-quit. This is
identity-restoration plumbing, not an emission trigger.

| Event | Authority | Action | Notes |
|---|---|---|---|
| `ProperSave.SaveFile.OnGatherSaveData` | host | persist `startTimeUtc` | At save time, write our captured `startTimeUtc` into the save's modded-data dictionary under `"me.byjp.pesos.ror2.startTimeUtc"`. See [`atproto-records.md`](atproto-records.md#bridging-the-propersave-gap). |
| `ProperSave.Loading.OnLoadingStarted` | host | restore `startTimeUtc` | On resume, read it back via `CurrentSave.GetModdedData<DateTime>(...)` and use it in place of `Run.instance.GetStartTimeUtc()` when deriving the rkey. |

Soft dep: the subscriptions are registered only if the ProperSave
assembly is loaded. Without it, vanilla RoR2 has no save-and-quit, so
nothing breaks.

[propersave]: https://github.com/KingEnderBrine/-RoR2-ProperSave

### Periodic safety net

There's **no vanilla autosave callback** in RoR2 (this is the biggest
divergence from StS2). The throttled timer described in [Emission policy](#emission-policy)
fills that gap. Implementation: a coroutine on `RoR2Application.instance`
that wakes every 5 s, checks dirty + throttle, and either emits or sleeps.

Alternative: `IL.RoR2.Run.FixedUpdate` IL hook gating on
`Time.unscaledTime - lastEmit > throttleSeconds`. Lower overhead than a
coroutine but harder to debug. Default to the coroutine first.

## Multiplayer authority

RoR2 is host-authoritative. The host is the source of truth for kills, item
grants, stage transitions. Two implications:

1. **The host writes the canonical run record.** Only the host's `Run`
   instance has the fully populated `RunReport.statSheet` post-game-over. The
   host's mod is responsible for the final write.
2. **Client peers can write their own perspective.** Each client peer's PDS
   record reflects *their* point of view (their items, their character,
   their kills). The web-side reader joins multiple peer records by the
   shared `seed + runStartTimeUtc` rkey derivative.

For v0: ship as a **client-side mod** (every peer writes its own record;
multi-write means everyone's PDS has their own record for the same run, and
the web app stitches them together via the `allies[]` list). This matches the
sts2.at model and avoids the host-only install constraint.

`R2API.NetworkCompatibility` can mark the mod as not-affecting-networking so
mixed installs work in a lobby.
